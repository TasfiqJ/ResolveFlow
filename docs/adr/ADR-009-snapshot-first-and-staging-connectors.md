# ADR-009: Use snapshot-first public mode and disabled real connectors

**Status:** Accepted
**Date:** 2026-07-21

## Context

The public product must be useful during provider outages and must not expose write credentials or an open-ended attack/cost surface. Slack/Jira credentials are unavailable by default. The project still needs truthful adapter depth and a path to bounded live evidence.

## Options considered

1. Require live Cohere, Slack, Jira, API, worker, and database for the public demo.
2. Publish only static screenshots/video disconnected from the production path.
3. Publish content-addressed sanitized run snapshots generated by the production path; keep optional bounded local/staging live mode and disabled real connector adapters.

## Decision

Choose option 3. Every public scenario loads a recorded snapshot first and labels build, commit, model policy, corpus, date, and provenance. Public inputs are predefined. Slack/Jira adapters have typed contracts and sanitized fixtures but remain disabled unless credentials already exist and the active task explicitly authorizes the exact use.

## Consequences

- GitHub Pages requires no API, database, provider, or connector credential.
- A matching snapshot is the honest fallback for unavailable/budget-exhausted live runs.
- Recorded output must never be labeled live; live usage and cost are reported only from actual traces.
- Public startup/build scans fail on write credentials or secret material.

## Rejected alternatives

- Live-only public mode is brittle, costly, and expands abuse risk.
- Screenshot-only proof cannot demonstrate trace integrity or production-path provenance.

## Reversal trigger

A later deployment has verified zero-cost/approved operations, strong rate limits, no write credential, and an availability/security reason to make live execution primary. Recorded fallback and provenance remain required.
